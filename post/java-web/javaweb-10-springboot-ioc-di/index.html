分层解耦#

在开发中，我们往往遵循一个原则——"单一职责原则"

即，一个类、一个方法、一个模块，只负责一件事。
一个变化的原因不要超过一个。

这样的代码：

  1. 可维护性更换 → 改一个功能不会影响其他功能
  2. 可复用性强 → 类职责越明确，越容易复用
  3. 代码可测试性提高 → 一个类只处理一件事，单元测试更简单
  4. 降低耦合度 → 模块之间独立，不相互拖累

三层架构#

✅ 三层架构是什么？

三层架构包括：

  • controller：控制层，接收前端发送的请求，对请求进行处理，并响应数据。
  • service：业务逻辑层，处理具体的业务逻辑。
  • dao：数据访问层（Data Access Object）（持久层），负责数据访问操作，包括数据的增、删、改、查。

      package com.javaweb.service;

import com.javaweb.dao.DeptDaoImpl;
import com.javaweb.entity.Dept;

import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;
import java.util.List;

public class DeptServiceImpl {
    private DeptDaoImpl deptDao = new DeptDaoImpl();
    public List<Dept> queryDeptList() {
        //    1.调用dao层方法
        List<String> lines = deptDao.queryDeptList();
        //    2.对原始数据进行处理，封装部门数据
        List<Dept> deptList = lines.stream().map((line) -> {
            String[] parts = line.split(",");
            Integer id = Integer.valueOf(parts[0]);
            String name = parts[1];
            LocalDateTime updateTime = LocalDateTime.parse(parts[2], DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
            return new Dept(id, name, updateTime);
        }).toList();
        return deptList;
    }

}

面向接口编程#

假如我们想换一套业务逻辑，需要怎么做？
在传统做法中，我们往往需要直接修改原有类的代码，把旧的逻辑替换成新的逻辑。这样做的问题是：一改就可能影响整个项目，风险极高，也不利于系统维护。

为了提高我们的可拓展性，我们需要采用面向接口编程的思想。 具体做法是：

  1. 先定义一个接口（Interface），只声明功能，不关心具体实现。
  2. 业务逻辑、数据访问等模块都基于接口编写，而不是依赖具体类。
  3. 当我们想更换新的业务逻辑时，不需要修改任何原有代码，只需要：
  • 新写一个实现该接口的类
  • 通过依赖注入或配置选择新的实现即可

这样，扩展 = 添加新的实现类
而不是修改旧代码，从而提升了系统的 可维护性、可扩展性和稳定性。

这也是最符合 开闭原则（OCP） 的

对扩展开放，对修改关闭。

不难发现现在我们的代码中使用了"面向接口+多态"的写法。 用接口声明、用实现类实例化，可以让 上层模块 只依赖“抽象的服务能力”，而不是具体的实现类。

有没有发现？以前云里雾里的"多态“此刻竟也明悟了。

分层解耦#

  • 耦合：衡量软件中各个层/模块之间的依赖、关联的程度
  • 内聚：软件中各个功能模块内部的功能联系
  • 软件设计原则：高内聚低耦合

现在我们的程序其实是强耦合（高耦合）的，因为它存在调用对象的实例化。 如何对它进行解耦呢？需要用到springboot的 控制反转 和 依赖注入

  • 控制反转：Inversion Of Control，简称IOC。对象的创建控制权由程序自身转移到外部（容器），这种思想称为控制反转
  • 依赖注入：Dependency Injection，简称DI。容器为应用程序提供运行时，所依赖的资源，称之为依赖注入。
  • Bean对象：IOC容器中创建、管理的对象，称之为Bean

IOC详解#

  • 要把某个对象交给IOC容器管理，需要在对应的类上加上如下注解之一：
    注解 说明 位置
    @Component 声明bean的基础注解 不属于以下三类时，用此注解
    @Controller @Component的衍生注解 标注在控制层类上
    @Service @Component的衍生注解 标注在业务层类上
    @Repository @Component的衍生注解 标注在数据访问层类上（由于和mybatis整合，不常用

信息

声明bean的时候，可以通过注解的value属性指定bean的名字，如果没有指定，默认为类名首字母小写。

前面声明的bean的四个注解，想要生效，还需要被组件扫描注解@ComponentScan扫描。
该注解虽然没有显示配置，但是实际上已经包含在了启动类声明注解@SpringBootApplication中，默认 扫描的范围是启动类所在包及其子包

DI详解#

  • @Autowired注解，默认是按照 类型 进行，如果存在多个类型相同的bean，将会报出如下错误

方案一：@Primary#

当容器中有多个同类型 Bean 时，可以通过在其中一个 Bean 上标注 @Primary，告诉 Spring 容器默认注入这个 Bean：

@Service
@Primary
public class UserServiceImpl1 implements UserService {
    // 实现方法
}

@Service
public class UserServiceImpl2 implements UserService {
    // 实现方法
}

然后在注入时，即使有多个实现，Spring 也会选择标记了 @Primary 的 Bean：

@Autowired
private UserService userService; // 自动注入 UserServiceImpl1

方案二：使用@Qualifier#

当存在多个同类型 Bean 时，如果你不想使用 @Primary，也可以通过 @Qualifier 指定具体要注入的 Bean 名称：

@Service("userService1")
public class UserServiceImpl1 implements UserService {
    // 实现方法
}

@Service("userService2")
public class UserServiceImpl2 implements UserService {
    // 实现方法
}

然后在注入时，使用 @Qualifier 指定 Bean 名称：

@Autowired
@Qualifier("userService2")
private UserService userService; // 注入 UserServiceImpl2

信息

⚠️ 注意：@Qualifier 的名称必须和 Bean 的名称一致，通常默认是类名首字母小写，或者使用 @Service(“自定义名称”) 指定。

方案三：使用@Resource#

@Resource 是 JSR-250 提供的注解，等价于@Autowired+@Qualifier，Spring 也支持它，用于注入 Bean。：

@Service("userService1")
public class UserServiceImpl1 implements UserService {
    // 实现方法
}

@Service("userService2")
public class UserServiceImpl2 implements UserService {
    // 实现方法
}

按名称注入

@Resource(name = "userService2")
private UserService userService; // 注入 UserServiceImpl2

不指定名称（按属性名注入）

@Resource
private UserService userService1; // 会注入名称为 userService1 的 Bean
