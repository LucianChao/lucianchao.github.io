Maven#

什么是Maven？#

Maven是一款用于管理和构建Java项目的工具，是apache旗下的一个开源项目。

信息

Apache 软件基金会，成立于1999年7月，是目前世界上最大的最受欢迎的开源软件基金会，也是一个专门为支持开源项目而生的非盈利组织。
开源项目：https://www.apache.org/index.html#project-list

Maven的作用#

  1. 依赖管理 方便快捷的管理项目依赖的资源（ jar包）
  2. 项目构建 标准化的跨平台（Linux、Windows、Maxos）的自动化项目构建方式
  3. 统一项目构建 提供标准、统一的项目结构

Maven概述#

介绍#

  • Apache Maven是一个项目管理和构建工具，它基于项目对象模型( POM——Project Object Model )的概念，通过一小段描述信息来管理项目的构建
  • 作用：
    • 方便的依赖管理
    • 统一的项目结构
    • 标准的项目构建流程
  • 官网：https://maven.apache.org/

仓库：用于存储资源，管理各州jar包

  • 本地仓库：自己计算机上的资源目录
  • 中央仓库：由Maven团队维护的全球唯一的仓库。仓库地址：https://repo1.maven.org/maven2/
  • 远程仓库（私服）：一般由公司团队搭建的私有仓库

安装#

  • 安装步骤：
    1. 解压 apache-maven-3.9.11-bin.zip
    2. 配置本地仓库：修改 conf/settings.xml中的 <localRepository>为一个指定目录
       <localRepository>E:\Devlop\Tools\apache-maven-3.9.11\repo</localRepository>
    1. 配置阿里云私服：修改 conf/settings.xml 中的<mirrors>标签，为其添加如下子标签：
    <!-- 新版本 -->
<mirror>
   <id>aliyunmaven</id>
   <mirrorOf>*</mirrorOf>
   <name>阿里云公共仓库</name>
   <url>https://maven.aliyun.com/repsitory/public</url>
</mirror>

<!-- 旧版本 -->
<mirrors>
   <id>alimaven</id>
   <name>aliyun maven</name>

   <mirrorOf>central</mirrorOf>
</mirrors>
    1. 配置JDK版本：修改 conf/settings.xml 中的profile环境为JDK21（可以不配）
    2. 配置环境变量：MAVEN_HOME为maven的解压目录，并将其bin目录加入PATH环境

Maven坐标#

  • 什么是坐标？
    • Maven项目中的坐标是 资源（jar）的唯一标识，通过该坐标可以唯一定位资源位置
    • 使用坐标来 定义项目 或 引入项目中需要的依赖。
  • Maven坐标的主要组成
    • groupId：定义当前Maven项目隶属组织名称（通常是域名反写，例如：com.itheima）
    • artifactId：定义当前Maven项目名称（通常是模块名称，例如order-service、goods-service）
    • version：定义当前项目版本号
      • SNAPSHOW：功能不稳定，尚处于开发中的版本，即 快照版本
      • RELEASE：功能趋于稳定，当前停止更新，可以用于发行的版本

依赖管理#

依赖配置#

  • 依赖：指当前项目运行所需要的jar包，一个项目可以引入多个依赖。
  • 配置：
    1. 在 pom.xml 中编写 <dependencies>标签
    2. 在<dependencies>标签中，使用<dependency>引入坐标
    3. 定义坐标的 groupId, artifactId, version
    4. 点击刷新按钮，引入最新加入的标签

信息

  • 如果引入的依赖在本地仓库不存在，将会连接远程仓库/中央仓库，然后下载依赖。（这个过程会比较耗时，请耐心等待）
  • 如果不知道依赖的坐标信息，可以到 https://mvnrepository.com/ 中搜索

依赖传递#

  • 依赖具有传递性
    • 直接依赖：在当前项目中通过配置建立的依赖关系
    • 间接依赖：被依赖的资源如果依赖其他资源，当前项目间接依赖其他资源

看完这些，我们可以思考一个问题。
上面的示例只有三个模块，关系非常简单。

但在真实开发中，我们往往会使用大量第三方库，而每个库又会继续依赖其他库。随着依赖链越来越长，就很容易出现问题。

  1. 引入了本不需要的依赖
    某些依赖传递你根本用不到，但它被“强制”带进了项目中。这些依赖不仅会让项目臃肿不堪，还可能导致安全风险或版本冲突。
  2. 版本冲突（最典型的问题）
    比如：
    • A依赖B（B依赖commons-io 2.6）
    • 同时A又依赖D（D依赖commons-io 2.4）
      这样两个版本同时出现就会引发不兼容的问题。
  3. 无法控制依赖来源

为了解决这些“强制性依赖带来的副作用”，Maven提供了一个非常重要的机制：依赖排除（Dependency Exclusion）

排除依赖#

指主动断开依赖的资源，被排除的资源无需指定版本

  • 可以在pom.xml中的<dependency>中配置<excliusions>中配置<excliusion>排除依赖

  • 也可以下载并使用Maven Helper插件

    下载安装好插件并重启IDEA后，当我们打开一个pom.xml文件时，就可以在下方看到文件的两种表现形式 → 我们选择 Dependency Analyzer（依赖关系分析器） → 找到我们想要排除的依赖 → 右键 → 选择 Exclude

    此时会自动生成<excliusion>标签，回到文本视图点击刷新按钮即可

生命周期阶段

  • clean：移除上一次构建生成的文件

单元测试#

测试#

测试是一种用来促进鉴定软件的正确性、完整性、安全性和质量的过程

测试分为四个阶段：单元测试、集成测试、系统测试、验收测试

  • 单元测试（白盒测试）
    • 介绍：对软件的基本组成单位进行测试，最小的测试单位
    • 目的：检验软件基本组成单位的正确性
    • 测试人员：开发人员
  • 集成测试（灰盒测试）
    • 介绍：将已分别通过测试的单元，按设计要求组合成系统或子系统，再进行测试
    • 目的：检查单元之间的协作是否正确
    • 测试人员：开发人员
  • 系统测试（黑盒测试）
    • 介绍：对已经集成好的软件系统进行彻底的测试
    • 目的：验证软件系统的正确性、性能是否满足指定的要求
    • 测试人员：测试人员
  • 验收测试（黑盒测试）
    • 介绍：交付测试，是针对用户需求、业务流程进行的正式的测试
    • 目的：验证软件系统是否满足验收标准
    • 测试人员：客户/需求方

测试方法一般有三种：白盒测试、黑盒测试 以及 灰盒测试

  • 白盒测试
    • 清楚软件内部结构、代码逻辑
    • 用于验证代码、逻辑正确性
  • 黑盒测试
    • 不清楚软件内部结构、代码逻辑
    • 用于验证软件的功能、兼容性、验收测试等方面
  • 灰盒测试
    • 结合了白盒测试和黑盒测试的特点，既关注软件的内部结构又考虑外部表现（功能）

单元测试#

单元测试：就是针对最小的功能单元（方法），编写测试代码对其正确性进行测试。

  • JUnit：最流行的Java测试框架之一，提供了一些功能，方便程序进行单元测试（第三方公司提供）

快速入门#

使用JUnit，对UserService中的业务方法进行单元测试，测试其正确性

  • Userservice.java 
    package org.example;

import java.time.LocalDate;
import java.time.Period;
import java.time.format.DateTimeFormatter;

public class UserService {
    /**
     * 给定一个身份证号，计算该用户的年龄
     *
     * @param idCard 身份证号
     */
    public Integer getAge(String idCard) {
        String birthday = idCard.substring(6, 14);
        LocalDate parse = LocalDate.parse(birthday, DateTimeFormatter.ofPattern("yyyyMMdd"));
        return Period.between(parse, LocalDate.now()).getYears();
    }

    /**
     * 给定一个身份证号，计算该用户的性别
     *
     * @param idCard 身份证号
     */
    public String getGender(String idCard) {
        return Integer.parseInt(idCard.substring(16, 17)) % 2 == 1 ? "男" : "女";
    }
}

常见注解#

注解 说明 备注
@Test 测试类中的方法用它修饰才能成为测试方法，才能启动执行 单元测试
@BeforeEach 用来修饰一个实例方法，该方法会在 每一个 测试方法执行之前执行一次 初始化资源（准备工作）
@AfterEach 用来修饰一个实例方法，该方法会在 每一个 测试方法执行之后执行一次 释放资源（清理工作）
@BeforeAll 用来修饰一个静态方法，该方法会在所有测试方法执行之前 只执行一次 初始化资源（准备工作）
@AfterAll 用来修饰一个静态方法，该方法会在所有测试方法执行之后 只执行一次 释放资源（清理工作）
@ParameterizedTest 参数化测试的注解（可以让单个测试运行多次，每次运行时仅参数不同） 用了该注解，就不需要@Test注解了
@ValueSource 参数化测试的参数来源，赋予测试方法参数 与参数化测试注解配合使用
@DisplayName 指定测试类、测试方法的显示名称（默认为类名、方法名）
  • @ParameterizedTest与@ValueSource示例： 
    package com.javaweb;

import org.example.UserService;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.params.ParameterizedTest;

/**
 * 测试类，测试UserService业务方法
 */
public class UserService2Test {
    @ParameterizedTest
    @ValueSource(strings = {"101010200304201133", "101010200304201134", "101010200304201111"})
    public void testGetGender(String id) {
        String gender = new UserService().getGender(id);
        System.out.println("gender = " + gender);
    }
}

断言#

JUnit提供了一些辅助方法，用来帮我们确定被测试的方法是否按照我们预期的效果正常工作，这种方式称为 断言

提示

上述方法 形参 中的最后一个参数 msg，表示错误提示信息，可以不指定（有对应的 重载方法）

在上面的测试用例中我们是通过身份证号后两位的奇偶性来获取性别的
奇数为男性，偶数为女性

return Integer.parseInt(idCard.substring(16, 17)) % 2 == 1 ? "男" : "女";

假如我们的程序写错了，比如说把偶数判断为男性，奇数判断为女性。

return Integer.parseInt(idCard.substring(16, 17)) % 2 == 1 ? "女" : "男";

那么测试方法虽然可以正常运行，但是结果却是错误的。
测试结果也显示通过

如果我们加上 断言，提前对结果进行预测，一旦结果不符合断言，测试就会不通过

这里我们以assertEquals断言方法为例
assertEquals(Object exp, Object act, String msg)
assertEquals(期待值，真实结果，错误信息)

package com.javaweb;

import org.example.UserService;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;

import static org.junit.jupiter.api.Assertions.assertEquals;

/**
 * 测试类，测试UserService业务方法
 */
@DisplayName("测试用户信息类")
public class UserService3Test {
    @DisplayName("测试性别")
    @Test
    public void testGetGender() {
        String gender = new UserService().getGender("101010200304201133");
//        System.out.println("gender = " + gender);
        assertEquals("男", gender,"性别获取错误");
    }
}

依赖范围#

依赖的jar包，默认情况下，可以在任何地方使用。可以通过<scope>...</scope>设置其作用范围

注释

一般会自动生成<scope>...</scope>标签，大部分情况下无需手动调整

作用范围：

  • 主程序范围有效（main文件夹范围内）
  • 测试程序范围有效（test文件夹范围内）
  • 是否参与打包运行（package指令范围）
scope值 主程序 测试程序 打包（运行） 范例
compile（默认） Y Y Y log4j
test - Y - junit
provided Y Y - servlet-api
runtime - Y Y jdbc驱动
示例#
<dependency>
  <groupId>org.junit.jupiter</groupId>
  <artifactId>junit-jupiter</artifactId>
  <version>5.8.2</version>
  <scope>test</scope>
</dependency>

依赖未正确下载#

下载依赖时，依赖未正确下载，造成右侧Maven面板中的依赖报红，再次reload重新加载也不会下载。

  • 产生原因：由于网络原因，依赖没有下载完整导致的，在maven仓库中生产力xxx,lastUpdated文件
  • 解决方案：
    1. 根据maven依赖坐标，找到仓库中对应的xxx,lastUpdated文件删除，重新加载项目即可
    2. 在仓库目录下，通过命令( del /s *.lastUpdated )批量递归删除指定目录下的xxx,lastUpdated文件，重新加载项目即可
