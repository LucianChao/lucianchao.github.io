从 extend_xx 到 partial#

在之前，我为我的博客加功能都是直接写在 extend_xx 中的，今天正打算写个新功能，忽然觉得这一坨代码开始有点 💩 山的感觉了，于是决定对功能注入逻辑的重构，一方面提高功能的可维护性，博客的可拓展性，另一方面也让逻辑更加清晰，看着更加清爽

原理解析#

一般我们加新功能都是加在 extend_footer.html 和 extend_head.html 中的这两个文件是 papermod 留给我们的自定义功能接口
像之前我为博客加的 fancybox 就是如此做的

  • extend_footer.html 
    <!-- Fancybox 全局脚本 -->
{{ if .IsPage }}
<link
  rel="stylesheet"
  href="/lib/fancybox/fancybox.css"
/>
<script src="/lib/fancybox/fancybox.umd.js"></script>
<script>
  document.addEventListener("DOMContentLoaded", function () {
    Fancybox.bind("[data-fancybox]", {// 绑定所有 data-fancybox 图片
      animated: true,                  // 启用动画效果
      showClass: 'fancybox-fadeIn',    // 打开动画
      hideClass: 'fancybox-fadeOut',   // 关闭动画
      dragToClose: true                 // 启用拖拽关闭
    });
  });
</script>
{{ end }}

而现在我们只要把这部分代码拆分出去，然后再把拆分出去的新文件导入进 extend_footer.html ， extend_head.html 中的代码也是同理。

文件详解#

首先我们先搞清楚 extend_footer.html 、 extend_head.html 这俩文件的区别

文件 注入位置 执行时机 本质作用
extend_head.html <head> 页面解析前 告诉浏览器“页面长什么样”
extend_footer.html </body> 前 DOM 已加载 告诉页面“怎么动”

因为它们的特性和执行时间不同，所以一般把影响页面结构、样式的代码放在 extend_head 中（如 css、 SEO、字体）。
影响页面交互、行为、加载后逻辑的放在 extend_footer 中（如 js、 DOM 相关逻辑、媒体处理、页面资源处理）

  3. 新建文件 fancybox.html，并把 extend_foot 中相关代码进行迁移

      <link
 rel="stylesheet"
 href="/lib/fancybox/fancybox.css"
/>
<script src="/lib/fancybox/fancybox.umd.js"></script>
<script>
 document.addEventListener("DOMContentLoaded", function () {
   Fancybox.bind("[data-fancybox]", {// 绑定所有 data-fancybox 图片
     animated: true,                  // 启用动画效果
     showClass: 'fancybox-fadeIn',    // 打开动画
     hideClass: 'fancybox-fadeOut',   // 关闭动画
     dragToClose: true                 // 启用拖拽关闭
   });
 });
</script>
  4. 在 extend_footer.html 中引入 fancybox.html
    • layouts\partials\extend_footer.html 
      {{ if .IsPage }}

<!-- Fancybox 全局脚本 -->
{{ partial "media/fancybox.html" .}}

{{ end }}
