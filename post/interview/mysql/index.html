面试题汇总#

  1. 数据库优化经验或慢SQL优化经验

    1. 表结构上的设计优化

      当看到SQL数据连接的表超过5张的时候，基本上可以考虑从库表结构上思考优化，一般情况是：

      • 业务的不必要拆分

      例如：新闻表（新闻id，标题，作者）；新闻详情表（新闻id，图片，发布时间，来源，内容）
      如果再App端需要展示新闻列表需要：（新闻id，标题，图片），这时需要从两个海量数据表中进行关联查询才能获得图片，因为新闻表中不存在图片的字段只能关联查询。
      上述的业务是否可以不拆分？或者拆分更合理一些。
      解决方案：表设计不合理，将图片字段放到新闻表，或者两张表合二为一。

      • 太过于规范的三范式设计
        例如：查询一个学生时什么学院什么专业，需要查询4张表，如果查询专业频繁的话，非常耗时且没必要。
        解决方案：反三范。
        可以在班级表中添加一个专业的中文名（冗余字段），减少对于专业表的连接。
      • 业务表、字段的缺失：旧业务没有考虑过新业务缺失的内容，从而需要关联非常多表影响性能。

      例如：积分系统一开始没有考虑到排行榜的数量问题，每次统计月度排行榜时都需要实时计算（利用积分记录表根据用户分组统计某个时段积分值再排名），上线时，进行系统整体压测，发现首页加载速度有一定延迟，定位到几个查询比较耗时，排行榜便是其中之一。 改造：新增赛季积分表定时统计积分信息到该表，需要时直接查询

    2. 索引优化

      • 表的主键、外键必须有索引
      • 数据量大的表应该有索引
        • 对于常用的查询条件创建索引
      • 经常与其他表进行连接的表，在连接字段上应该建立索引
      • 经常出现在where字句中的字段，特别是大表的字段，应该建立索引
      • 索引应该建在区分度高的字段上（sex性别，status状态等这种就不适合，但适当可创建联合索引达到覆盖索引效果）
        • 如果这些区分度不高的字段需要在覆盖索引的时使用，也可以创建联合索引
      • 索引应该建立在小字段上，对于大的文本字段甚至超长字段，不要建索引
        • 如果text、varchar非常长、json等类型的字段不建立索引
      • 频繁进行数据操作的表，不要建立太多索引
        • 会重新建立索引
      • 删除无用的索引，避免对执行计划造成负面影响

        删除数据使用delete
        删除数据库对象（tb_表、idx_索引、fun_函数、pro_存储过程、v_视图）使用drop

    3. sql语句优化

      • select语句务必指明列的名称（避免直接使用select *）
        • 使用*的话，很可能出现回表，影响性能
      • sql语句要避免造成索引失效的写法
        • 不符合最左匹配原则
        • where条件中使用函数
        • 使用运算符
        • 使用类型的隐式转换（表中存varchar，查询数字xxx，应该查询’xxx'
      • SQL语句中IN包含的值不应该过多，可以的话使用between、and代替
      • 如果排序字段没有用到索引，就尽量少排序
      • 如果限制条件中其他列没有索引，尽量少用or
      • 尽量使用union all（不去重）代替union
      • 不建议使用%前缀模糊查询（最左匹配）
      • 避免在where子句中对列进行表达式操作
      • Join优化能用inner join就不用left join、right join小表驱动大表

    4. 缓存优化 为了提高查询速度，可以通过不同的方式去缓存我们的结果从而提高响应效率，使用redis就是最佳实践之一。

    5. 读写分离
      如果数据库的使用场景读多写少的时候，为了避免写操作造成的性能影响。可以采用读写分离的架构，读写分离解决的是数据库的写入影响查询效率。
      读写分离的基本原理是让主数据库处理事务性增、改、删操作，而从数据库处理select查询操作。利用binlog日志把事务性操作导致的变更同步到集群中的从数据库。

    6. 分库分表 数据量越来越大，单体数据库无法满足要求，可以考虑分库分表。
      两种拆分方案：

      • 垂直拆分：（分库）将业务细化，不同的小业务专门用一个库来维护
      • 水平拆分：（分表）单个表数据太多，可以将该表拆分成多个。例如赛季排行榜相关业务。

      分库分表常用中间件：MyCat（数据库中间件）、Sharding-JDBC

  2. MySQL的常见存储引擎有哪些？
    • InnoDB
    • MyISAM
    • MySQL 常见的存储引擎主要有 InnoDB、MyISAM、Memory、Archive、CSV、NDB 等。其中最常用的是 InnoDB，它支持事务、行锁和外键，是 MySQL 5.5 之后的默认存储引擎；MyISAM 不支持事务，使用表锁，适合读多写少的场景；Memory 引擎数据存储在内存中，查询速度很快，但重启会丢失数据；Archive 主要用于历史数据归档。

MySQL怎么存数据？#

MySQL数据保存在磁盘上，数据处理发生在内存中，数据页是磁盘与内存之间交互的基本单位，也是MySQL管理存储空间的基本单位，大小默认为16KB

数据页#

数据页：存放有多条完整记录的存储空间

  • 如果没有数据页：需要一条一条记录从磁盘读取到内存，频繁的IO，影响性能
  • 如果有数据页：将这16KB的记录组成为一个页。运用程序局部性原理。加载一页进行一次IO。与没有利用页的方式比较，大大的降低了磁盘IO

    信息

    程序局部性原理：程序在某一段时间内，只会集中访问一小部分数据或指令。

目录页#

有了页之后，只是减轻了部分IO，加载到内存后还是要一条条比较，除非查询到了数据不然会一直加载所有页，这样依然不够快，如果能知道要查找的数据在哪一页的话，就可以直接加载对应页号的数据。
因此，之江每个页的页号和比较的数据作为一条页号记录存入一页中。如图：

把针对页生成的目录对应的页称为目录页，而之前存储完整记录的页称为数据页

索引#

目录页多了之后也需要提高对于目录的查询，如果目录页之间也可以根据一定的形式进行组织。保护用户数据和目录项。目录项中保存页的记录中主键的最小值与页号，从而保证下一个数据页中记录的主键值大于上一个页中记录的主键值。可形成如下：

上图呈现的也就是索引，是一种数据结构，用于加速查询
MySQL中是由存储引擎实现索引，InnoDB存储引擎索引基于B+树实现，也就是MySQL中每一个索引都是一棵B+树。每个表的数据也是存在B+树中。

索引是增加了目录页及其之间关系的存储，会占用存储空间。所以索引是一种以空间换时间思想的具体实现 索引中保存的数据与索引的类型有关 索引可以分为两种类型：

  • 聚簇索引，主键索引。叶子节点中保存主键值 + 对应的完整行记录，目录项中保存主键最小值 + 页号。InnoDB属于索引组织表，每张表都有聚簇索引，因此表必须有主键，表中行的物理顺序与索引的逻辑顺序相同。
  • 非聚簇索引：二级索引，在非主键的其他列上建的索引，叶子节点中保存索引列的值 + 对应的主键值，目录项中保存索引列最小值 + 对应的主键值 + 页号

B+树#

  • B+树是一种n叉排序树，每个节点通常有多个子节点。
  • 最顶层的目录页称为根节点，最下层的存储完整记录的数据页称为叶子节点
  • 用户数据保存在叶子节点，目录项保存在非叶子节点，每个节点可能保存多个页
  • 最上面的节点称为根节点，根节点的地址保存在内存的数据字典中
  • B+树的深度一般控制在3层以内，因此定位到单挑记录不超过3次IO
  • 目录项和记录是排好序的，就可以通过二分法来快速定位查找

MySQL B+树深度超过4层，性能有非常大影响，所以一般控制在3层

示例：

如果B+树最高3层，如下的表最多能存多少条记录：

CREATE TABLE `tb_user` (
    `id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
    `name` varchar(20) DEFAULT NULL COMMENT '姓名',
    `age` tinyint(3) unsigned DEFAULT NULL COMMENT '年龄',
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='用户表'

具体存储结构示例如下：

  • 第一层：在这层只存储目录数（主键 + 页号指针）。指针在InnoDB中为6字节，本示例中主键为bigint，占8字节；得出： 每条目录大小 = 主键+页号指针 = 8+6 = 14字节
  • 第二层：存储目录数，因为第一层最多可以存储1170个目录数，相当于第二层有1170个页号，而每页也是16kb大小，它也可以存储1170个目录树，也就是第二层的 总目录数 = 1170 + 1170 = 1368900
  • 第三层：叶子节点，每页存储数据，页大小依然是16kb，那么需要根据表结构计算出来，每条记录占多少字节。tb_user表的一行记录 = id + name +age 字段存储字节的大小 +指针大小 = 8 + (20*3 +1) + 1+6 = 76
    一页可以存储数据条数 = 16KB / 76 = (16*1024) / 76 =215
    那么一颗3层B+树存储的数据最多可达到：1170 * 1170 * 215 = 294,313,500

bigint占8字节
varchar根据存储的实际值的字符长度*每个字符占比+1(0~255个字符以内加1，255以上则+2)，所以20*3+1表示每个name的长度都是20个字符（这是最大值，不准确）。因为采用的是utf8编码，每个字符占3个字节。字符最大长度为20，所以在255内则加1。
tinyint占1字节

三个性能相关概念#

覆盖索引#

当索引中包含要查询的所有字段时，这个索引称为覆盖索引，性能非常好。

回表#

当普通二级索引中不包含要查询的所有字段时，就需要先通过二级索引查出主键索引，再通过主键索引查询二级索引中没有的其他列的数据，这个过程叫做回表。

索引下推（ICP）#

Using index condition：在利用索引进行条件1 进行查询的基础上再利用条件进行过滤（存储引擎层），最后才回表使用where的其他条件进行过滤（服务层）。
下推的意思是：ICP（优化器）尽可能的把index condition的处理从MySQL服务处下推到MySQL存储引擎层（索引）。并非全部WHERE条件都可以用ICP筛选，如果WHERE条件的字段不在索引中，还是要读取整表的记录到Server端做WHERE过滤

索引下推：用于优化二级索引从表中检索行的实现。条件过滤可以下推到存储引擎进行，先由索引元组（index tuple）根据查询条件进行过滤，满足条件的前提下才回表，否则跳过，相当于延迟加载数据行，因此ICP可以降低回表次数和IO次数

尽可能的利用索引查询。

联合索引#

示例：

/*ref: Using index condition：需要回表*/
explain select id,dep_id,age from employee where id > 200 and dep_id = 4;

/*创建联合索引，优化上述语句的回表，提升性能*/
alter table employee add index idx_dept_id_age(dep_id,age);

/*ref: Using where; Using index：不需要回表*/
explain select id,dep_id,age from employee where id > 200 and dep_id = 4;

最左匹配原则#

使用 联合索引（组合索引） 时，查询条件必须 从索引最左列开始连续匹配，否则索引无法完全使用。

CREATE INDEX idx_user ON user(name, age, city);
  • 使用索引name 
    SELECT * FROM user WHERE name = 'Tom';
  • 使用索引name+age 
    SELECT * FROM user WHERE name='Tom' AND age=20;
  • 使用索引name+age+city 
    SELECT * FROM user WHERE name='Tom' AND age=20 AND city='Beijing';
  • 部分使用索引
    • 只使用name 
      SELECT * FROM user WHERE name='Tom' AND city='Beijing';
      • 原因：跳过了age，所以city无法使用索引

分页加速#

假如要查询第1000页，每页1000条的员工数据（用户id、姓名、部门名称）

  • 方法一： 
    -- 耗时3s 365md
-- explain 后发现，只用到了一个dep_id的非唯一索引
select e.id,name,deptName from employee e join department d on e.dep_id=d.id limit 999000,1000;

-- 思考：索引覆盖-最快查询  114ms
select id from employee limit 999000,1000;

-- 查询用户信息 128ms
-- 先利用子查询查出id，只需要回表1000次
select e1.id,e1.name,e1.dep_id
from employee e1,(select id from employee limit 999000,1000) e2
where e1.id=e2.id;

-- 处理部门信息 143ms
select e.id,e.name,d.deptName
from(select e1.id,e1.name,e1.dep_id
     from employee e1,(select id from employee limit 999000,1000) e2
     where e1.id=e2.id) e, department d
where e.dep_id=d.id;
  • 方法二（不推荐）： 使用id限定方案，将上一页的id传递过来，根据id范围进行分页查询
    通过程序的设计，持续保留上一页的id，并且id保存自增且不跳号（条件苛刻，但效率很高）
select e.id, e.name, d.deptName
from employee e,department d
where e.dep_id = d.id and e.id>999000
limit 1000;
-- 999000就是前端传来的上一些最后一条id（最大或最小）

慢SQL&优化#

  1. 获取慢查询语句 mysql中可以设置慢查询的时间，查询的时长多久属于慢查询，默认10s，可以修改为3s等自定义参数。并且要开启慢查询以及慢查询的sql语句存放路径。
    可以通过查看这个慢查询存放的语句文件中，得知哪些语句是慢查询的SQL语句。
  2. 分析 explain + 慢查询语句，根据返回结果分析
  • type ALL/非ALL
  • key
  • ref
  • extra

如果type为ALL，则需要优化SQL语句，让它走索引。 3. 优化

  1. sql语句
    • 调整返回结果 减少回表
    • 尽可能的让子查询的条件都有对应的索引可以使用
    • 尽量使用覆盖索引
    • 是否减少了*的使用
    • 有没有哪些让索引失效的使用（函数、计算）
  2. 语句已经无法优化
    • 是否可以添加一些表的冗余字段
    • 有没有必要重新构建表结构和关系
    • 是否可以在程序中进行处理而减少表的链表耗时查询

三范式#

第一范式(1NF)#

字段必须是原子性，不可再分

一个字段只能存一个值，不能是集合、数组、多个值。

第二范式(2NF)#

非主键字段必须完全依赖主键
不能只依赖主键的一部分。

第三范式(3NF)#

非主键字段不能依赖其他非主键字段，消除传递依赖
